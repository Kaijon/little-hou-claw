# 內容摘要

**📌 來源**
- 類型：影片
- 標題：Inside Claude Code With Its Creator Boris Cherny
- 網址：[https://www.youtube.com/watch?v=PQU9o_5rHC4](https://www.youtube.com/watch?v=PQU9o_5rHC4)

**💡 核心概述**
這支訪談主要在談 Claude Code 為什麼會成功：核心不是把現有 IDE 做得更漂亮，而是持續追著模型能力往前走，並把產品設計成能跟著模型一起演化。Boris Cherny 強調，產品要以「六個月後的模型」來設計，而不是今天的模型；因此 Claude Code 會反覆重寫、快速試驗、從使用者回饋中迭代，甚至連 plan mode、sub-agents、CLAUDE.md 這些功能，都是從真實使用情境裡長出來的。

訪談也提到，隨著模型變強，開發者的工作型態正在改變：從手動寫碼，轉向寫規格、帶著代理做計畫、檢查結果與修正偏差。Anthropic 內部已經有許多工程、設計、PM、財務等角色都在用 Claude Code 工作，顯示這不只是編輯器工具，而是新的軟體生產方式。

**🔍 重點條列**
- Claude Code 的設計哲學是「不要為今天的模型設計」，而要為未來幾個月內會變強的模型設計。
- 產品大量依賴快速迭代：功能會一直被重寫、工具會持續更換，沒有什麼部分能維持半年不變。
- CLAUDE.md 的定位很克制：只放重複出現、真的有幫助的規則，太長反而該刪掉重來。
- plan mode 只是把「請先不要寫碼」這個意圖加進提示詞，但在實務上能大幅改善複雜任務的穩定度。
- sub-agents 被用來做平行除錯、查 log、查 code path，讓模型自己把工作拆開。
- 產品不是只服務工程師；Anthropic 內部很多非工程角色也在用同一套工具，因為它本質上是通用代理。
- Boris 認為未來的工作重心會從「寫程式」轉向「寫規格、跟使用者對話、指揮代理執行」。

**📊 關鍵數據與事實**
- 6 個月：產品設計時看的時間尺度。
- 約 80%：他自己多數 session 會先從 plan mode 開始。
- 約 150%：Claude Code 出現後，Anthropic 每位工程師產能成長幅度。
- 約 70%～90%：Anthropic 內部不同團隊的程式碼由 Claude 產出比例。
- 約 4%：公開 GitHub commits 來自 Claude Code 的引用數據。

**🎯 行動建議**
- 做 AI 工具時，先假設模型很快會變強，避免把產品綁死在今天的能力上。
- 把常見錯誤收進可重用的規則檔，但保持精簡，避免累積成負擔。
- 複雜任務先讓模型規劃，再讓它執行；必要時用多個 sub-agents 平行探索。
- 把重點放在使用者回饋與快速迭代，而不是一次把流程設計到完美。
